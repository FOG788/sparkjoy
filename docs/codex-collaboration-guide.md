# SparkJoy Codex コラボレーションガイド（ドラフト）

最終更新: 2026-03-22

この文書は、Codex と人間の開発者が **同じ前提で編集・レビューしやすくするための叩き台** です。  
初版のため、運用しながら随時更新してください。

---

## 1. このプロダクトの方向性（開発方針）

### 目指すもの
- SparkJoy は、ブラウザで即使える「タイピングに反応する遊び心付きエディター」を提供する。
- 静的サイト構成（`index.html` / `app.js` / `styles.css`）を活かし、**配布・起動のしやすさ**を最優先にする。
- 派手さよりも「すぐ動く・壊れにくい・挙動が説明可能」を重視する。

### 優先順位
1. **体験の一貫性**（説明文・UI表示・実装の整合）
2. **軽量性と単純性**（不要な依存や複雑な構成を増やさない）
3. **調整可能性**（音量やしきい値などをユーザーが調整できる）
4. **演出の楽しさ**（黄/赤ギミック、砂時計、効果音など）

### 非目標（現時点）
- 大規模なフレームワーク導入
- サーバーサイド前提の機能追加
- 高度な認証・マルチユーザー管理

---

## 2. 想定ユーザー

### 主な対象
- タイピングしながら雰囲気や演出を楽しみたい個人ユーザー
- 軽量なWebアプリを題材に、フロントエンド改善を試したい開発者

### 期待する利用シーン
- ローカルで素早く起動して使う（学習・気分転換・デモ）
- 設定を少し調整しながら自分好みの体験を作る

---

## 3. 実装時の基本原則（Codex / 人間共通）

### 3.1 変更の粒度
- 1つのPR（またはコミット）で扱う目的は原則1つに絞る。
- UI文言修正・ロジック変更・テスト追加を同時に行う場合は、関連性をコミットメッセージに明記する。

### 3.2 整合性チェック
- `index.html` の初期値と `app.js` の既定値は必ず一致させる。
- 説明書文言（UI/README/docs）と実際の挙動が矛盾しないようにする。
- 永続化項目（`localStorage` / cookie）は、読み書き双方のキー名・型を確認する。

### 3.3 体験優先の判断
- 新機能を足す前に、設定項目や文言を減らせないか検討する。
- 追加演出は「初見で理解できるか」「オフにできるか」を満たす。

## 4. 共著しやすくするための運用テンプレート

### 4.1 変更提案テンプレート（Issue/メモ用）

```md
## 背景
（なぜ必要か。困っていることは何か）

## 変更案
- [ ] UI
- [ ] ロジック
- [ ] テスト
- [ ] ドキュメント

## 受け入れ条件
- [ ] 期待挙動が再現できる
- [ ] 既存挙動を壊さない
- [ ] 説明文と実装が一致している
```

### 4.2 コミットメッセージの推奨フォーマット

```text
<type>: <変更の要約>

- 背景:
- 変更点:
- 確認方法:
```

`type` 例: `fix`, `docs`, `refactor`, `test`, `feat`

### 4.3 PR本文の推奨フォーマット

```md
## 目的

## 変更内容
- 

## 確認手順
1. 
2. 

## リスク/未対応
- 
```

## 5. Codex に依頼するときの書き方ガイド

### 5.1 良い依頼例
- 「`app.js` の cookie 読み取りを空白有無に依存しない形へ修正し、関連テストを追加して」
- 「`index.html` と `app.js` の既定値不一致を洗い出し、修正案を docs にまとめて」

### 5.2 依頼時にあると良い情報
- 期待する挙動（Before/After）
- 影響範囲（触ってよいファイル）
- 完了条件（テスト、目視確認、文言確認など）

### 5.3 避けたい依頼
- 目的が曖昧で成功条件がない依頼
- 一度に大きすぎる変更（UI刷新 + ロジック全面改修 + 新機能群）

## 6. 今後の拡張メモ（ドラフト）

- ユーザー向け説明の短縮版（クイックスタート）を `index.html` または別ページへ整理

- テスト拡充（既定値整合、永続化、表示文言の整合）
