# SparkJoy リポジトリ全体レビュー（オミット候補）

最終更新: 2026-03-11

## 結論（優先度つき）

### P0（先に直すべき）
1. **既定値の統一を継続監視する**
   - しきい値（黄/赤）とウォームアップ秒数は、UI 初期値と復元ロジック既定値を常に一致させる。
   - 今後の変更では、`index.html` の `value` と `app.js` の `load*` 既定値を同時更新する。

2. **説明書のしきい値説明を実装仕様に合わせた状態を維持する**
   - 実装は「入力速度がしきい値以上になると黄/赤ギミック」を発火する設計。
   - 説明文も同じ基準（速度上昇側）を維持し、逆方向の表現（速度低下）を混在させない。

### P1（機能整理の効果が高い）
3. **設定項目の段階表示（詳細設定の折りたたみ）を検討**
   - 「リアル度 / リバーブ / ジャム確率 / 黄赤しきい値 / ウォームアップ」まで常時表示され、初見ユーザーには調整負荷が高い。
   - MVP としては、`音量` と `エフェクトON/OFF` を主表示、残りは詳細設定に寄せる余地がある。

### P2（容量・体験の最適化）
4. **効果音サンプル数の縮小（4→1〜2）を検討**
   - `sounds/` に複数 MP3 を同梱し、起動後にまとめてデコードしている。
   - 静的配信の初期体験を軽くするなら、サンプル点数を絞るか、合成音のみ運用に寄せる選択が可能。

5. **「説明書」タブの分離検討**
   - 画面内に長文の説明書を持つ構成は、エディタUIとしては情報量過多になりやすい。
   - README か別ヘルプページへ切り出すと、編集画面をよりミニマルにできる。

## 補足（判断基準）
- 本レビューは「壊れている箇所の修正」ではなく、**機能削減・要素削減の候補出し**にフォーカス。
- 削除判断は次の優先順を推奨:
  1) 現行挙動と説明の不一致
  2) UXの学習コストを上げる設定
  3) 容量コストの高い演出資産
